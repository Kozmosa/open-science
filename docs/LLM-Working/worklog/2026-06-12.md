## Per-User Default Projects for All Users + Orphan-Task Guard

**User question → three-part solution.** User asked whether every registered user already had their own default project; code audit showed no — `<username>_default` is created only lazily via `get_or_create_user_default`, called solely in `POST /tasks` when `project_id` is empty, with no registration-time or backfill path. Worse, `ConvertToTaskDialog` sent a raw `<username>_default` string that the backend stored WITHOUT creating the project, producing orphan tasks. User specified the fix:

1. **Registration-time provisioning** — `POST /auth/register` now calls `project_service.get_or_create_user_default(username, owner_user_id)` right after `ensure_tenant_workspace`, mirroring how the Linux tenant user is provisioned.
2. **Backfill existing users** — new `ainrf.projects.backfill.backfill_user_default_projects()`, invoked idempotently at lifespan startup (after bootstrap-admin creation), guarantees every existing user — including the admin created directly in lifespan, which bypasses the HTTP hook — owns a default project. Logs the number created; failures are caught+logged so startup is never blocked.
3. **Anti-orphan guard** — `POST /tasks` now validates that an explicitly bound `project_id` exists in the registry (400 `Project not found` otherwise). Empty `project_id` still auto-resolves to the user default (created on demand). `ConvertToTaskDialog` now sends empty `project_id` so literature-to-task conversion routes through the backend resolver into the user default instead of a never-created raw string.

- Also genericized `_run_sync_in_lifespan` to `Callable[[], T] -> Awaitable[T]` so the startup backfill can return its `(created, skipped)` counts (ty was rejecting the `None`-typed helper).
- Backend tests touched by the new validation (5 tests POSTing `/tasks` with unregistered project ids like `proj-001`/`proj-token`/`proj-archive`) now seed the project via a `_seed_project` helper so task creation validates against a real registry entry; direct `service.create_task` callers are unaffected (validation is route-only).
