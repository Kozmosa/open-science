# OpenScience 文献追踪服务重设计

**Status:** Accepted product and service design; execution framework selection pending
**Date:** 2026-07-11
**Scope:** 文献追踪用户体验、持久化数据、每日检查、按需摘要、研究任务转化、统一后端 API 与迁移边界
**Supersedes:** 当本文与 [`2026-05-21-literature-tracking-design.md`](2026-05-21-literature-tracking-design.md) 或 [`docs/proposals/literature-fetch-refactor.md`](../../proposals/literature-fetch-refactor.md) 冲突时，以本文为准

## 1. 背景与结论

当前文献页面以“左侧订阅管理、右侧论文列表、按订阅触发 arXiv 抓取”为中心。这个模型可以支持最小功能，但没有围绕研究者真正的工作过程组织体验，也把文献获取、摘要、持久化和用户状态串成了一条脆弱链路。

本次重设计确认以下产品决策：

1. 文献页面的首页是“今日文献收件箱”，不是订阅列表。
2. 产品中的“订阅”统一改称“关注方向”；同一篇论文可以匹配多个关注方向，但只显示一份。
3. 摘要默认按需生成，只允许对少量高相关论文采用可配置的自动生成策略。
4. 每日文献收集由系统统一完成；手动检查必须复用相同的缓存、去重、限流和恢复流程。
5. 所有有业务意义的文献数据、用户状态和后台处理状态都必须持久保存，不能只存在于运行进程内存中。
6. 前端统一通过 `/api/literature` 接口组使用文献能力；旧接口只作为迁移期兼容入口。

新的产品模型是：

```text
创建关注方向
  → 系统统一收集文献源
  → 文献与版本进入本地文献库
  → 在本地匹配各用户的关注方向
  → 新论文进入今日收件箱
  → 用户阅读、收藏或忽略
  → 按需生成摘要
  → 精读或转成研究任务
```

## 2. 设计目标

### 2.1 用户体验目标

- 用户进入页面后首先知道“今天有什么新内容”，而不是先操作订阅配置。
- 用户能够区分“没有新论文”“正在检查”“检查部分完成”和“外部来源暂时失败”。
- 同一篇论文无论命中多少关注方向，都只出现一次，并展示全部匹配来源和原因。
- 用户不需要理解 arXiv 请求、分页、后台任务或缓存细节。
- 论文基础信息应尽快可见；摘要失败或延迟不能阻止论文进入收件箱。
- 页面上每个动作都具有稳定、可解释的结果，重复点击不会制造重复工作。

### 2.2 服务目标

- 从 arXiv 获得的有效结果必须先持久保存，再进行匹配、摘要或研究任务转化。
- 任一下游步骤失败时，只重试失败步骤，不重新消耗已经成功完成的上游调用。
- 系统可以证明某个查询和时间范围是否已经完整检查，而不是仅根据“数据库里已有一些论文”推测缓存完整。
- 每日自动检查、用户手动检查和首次关注方向补充检查必须走同一条后台流程。
- 生产和 staging 的数据、后台工作与外部调用额度必须完全隔离。
- 运行进程重启后，未完成工作可以从持久状态继续。

### 2.3 非目标

本文不决定 Redis Streams、Dramatiq、Celery、SQLite 工作队列或其他具体执行框架。后续选型必须服从本文定义的产品流程、数据边界和恢复语义。

首轮也不包含：

- 引用网络或相似论文扩散；
- 自动下载和长期保存全文 PDF；
- 默认抓取每篇论文的 HTML、PDF 首页或第三方代码链接；
- 为全部新论文自动生成 LLM 摘要；
- 邮件、Slack 等主动通知渠道；
- 面向公开互联网的文献搜索服务。

## 3. 产品信息架构

文献功能由四个主要用户界面组成：

1. 今日文献收件箱；
2. 论文详情；
3. 关注方向管理；
4. 更新记录。

“全部文献”“未读”“已收藏”和“有新版本”是文献列表的不同视图，不需要各自成为独立产品模块。

### 3.1 今日文献收件箱

这是 `/literature` 的默认页面。

页面顶部显示：

- 上次成功检查时间；
- 下次计划检查时间；
- 当前是否正在检查；
- 今日新增论文数量；
- 有新版本的论文数量；
- 未读与收藏数量；
- 是否存在部分完成或稍后重试的检查。

主列表提供以下视图：

- 今日更新；
- 未读；
- 已收藏；
- 有新版本；
- 全部文献。

可选筛选条件包括：

- 关注方向；
- arXiv 分类；
- 发布时间或更新时间；
- 是否已有摘要；
- 是否已关联研究任务。

页面只保留一个明确的外部更新动作：“检查最新文献”。它放在页面状态区域，不与普通列表刷新混用。

### 3.2 论文卡片

每篇论文只显示一张卡片。卡片默认包含：

- 标题；
- 作者；
- 首次发布时间与最后更新时间；
- 主要分类；
- abstract 摘要片段；
- 匹配的全部关注方向；
- 简短匹配原因；
- 新到、已读、已收藏、有新版本等状态；
- 摘要状态；
- 原文与 PDF 入口。

一级动作：

- 标记已读或未读；
- 收藏或取消收藏；
- 忽略；
- 打开详情。

二级动作：

- 生成摘要；
- 转成研究任务。

“转成研究任务”不再与“查看原文”同等突出。它属于用户完成初步筛选后的下一步。

### 3.3 论文详情

论文详情显示：

- 完整标题、作者、abstract 和来源链接；
- 当前版本与版本历史；
- 每个版本的发布时间、更新时间和内容变化提示；
- 匹配的关注方向和匹配原因；
- 阅读状态；
- 摘要内容及生成状态；
- 已关联的研究任务；
- “生成或重新生成摘要”和“创建研究任务”动作。

当论文出现新版本时，页面应明确区分：

- 论文首次进入文献库；
- 论文发布了新版本；
- 当前摘要对应旧版本，需要更新。

### 3.4 关注方向管理

关注方向是用户描述研究兴趣的入口，不直接等同于一次 arXiv 查询。

每个关注方向包括：

- 名称；
- 包含关键词；
- 排除关键词；
- 可选分类范围；
- 更新频率；
- 是否启用；
- 创建时间；
- 上次成功匹配时间；
- 当前状态。

创建或编辑时提供“预览匹配”能力。预览优先使用本地文献库，展示：

- 预计匹配数量；
- 最近的示例论文；
- 本地资料覆盖范围；
- 是否需要后台补充检查。

保存关注方向不应直接绕过统一检查流程请求 arXiv。若本地覆盖不足，系统创建一项后台补充检查，并立即返回已保存的关注方向。

关注方向状态包括：

- 等待首次检查；
- 正常关注；
- 已暂停；
- 需要调整条件。

### 3.5 更新记录

更新记录帮助用户理解系统是否真正完成了工作。

每条记录显示：

- 计划检查时间；
- 实际开始和结束时间；
- 覆盖的时间范围；
- 涉及的关注方向或分类；
- 取得、保存、匹配和新增论文数量；
- 使用缓存的情况；
- 完成、部分完成、稍后重试或失败状态；
- 对用户可理解的失败说明；
- 是否可以安全重试。

页面示例：

```text
7 月 11 日 03:00：已完成，新增 12 篇，有 3 篇发布新版本
7 月 10 日 03:00：部分完成，剩余分页将在稍后继续
7 月 9 日 03:00：arXiv 暂时不可用，当前仍显示上次结果
```

## 4. 核心用户流程

### 4.1 首次创建关注方向

1. 用户输入名称、关键词、排除词和可选分类。
2. 前端请求本地预览。
3. 系统返回本地已有的匹配示例与覆盖说明。
4. 用户确认保存。
5. 系统立即在本地匹配已有论文。
6. 如果本地资料不足，后台安排补充检查。
7. 用户无需停留等待，可以立即看到已有结果和“等待补充检查”状态。

### 4.2 每日自动检查

1. 系统在固定时间寻找需要更新的关注方向。
2. 将可以复用的文献源检查合并，避免按用户或关注方向重复请求。
3. 为本次检查确定固定的开始时间、结束时间和分页范围。
4. 逐页取得结果，每页成功后立即持久保存原始结果和论文版本。
5. 在本地将论文匹配到各用户的关注方向。
6. 将新论文或新版本加入对应用户的今日收件箱。
7. 只有全部必要分页完成后，本次时间范围才标记为完整覆盖。
8. 摘要不属于每日检查的完成条件。

### 4.3 手动检查最新文献

1. 用户点击“检查最新文献”。
2. 系统先判断最近是否已经完整检查相同范围。
3. 如果缓存仍然有效，直接返回最近结果。
4. 如果相同检查正在进行，返回同一个检查记录。
5. 只有确实存在覆盖缺口时才创建新的检查。
6. 前端展示持久化检查状态，而不是保持一个长连接等待全部结束。
7. 检查部分完成时，已获得的论文可以先展示。

### 4.4 按需生成摘要

1. 用户在卡片或详情页请求摘要。
2. 系统判断当前论文版本是否已有有效摘要。
3. 已有有效摘要时立即返回。
4. 已有相同请求正在处理时复用该请求。
5. 否则创建摘要工作记录。
6. 每篇或每个小批次成功后立即保存结果。
7. 失败只影响摘要状态，不影响论文、关注方向或每日检查状态。

### 4.5 转成研究任务

1. 用户选择研究方式、工作区、执行环境和其他任务配置。
2. 系统创建研究任务并建立论文关联。
3. 这两个动作对用户表现为一个完整操作。
4. 如果中间失败，系统保存“正在补完”或“创建失败”状态，不允许出现已经创建任务但文献页面永久不知道其存在的情况。
5. 论文详情可以展示一个或多个相关研究任务。

## 5. 持久化数据设计

所有影响用户体验、去重、恢复或额度判断的数据都必须入库。后台队列或缓存系统只保存可重建的传输信息。

### 5.1 数据分层

#### 公共文献数据

同一篇论文只保存一份：

- 论文身份；
- 标题、作者、分类和来源；
- 当前版本；
- 历史版本；
- abstract 与内容指纹；
- 原文和 PDF 地址；
- 首次获取和最后更新时间。

#### 用户私有数据

- 关注方向；
- 论文与关注方向的匹配关系；
- 匹配原因；
- 用户对论文的阅读、收藏和忽略状态；
- 用户何时首次看到论文；
- 研究任务关联。

#### 运行与恢复数据

- 每日或手动检查记录；
- 固定检查范围；
- 分页进度；
- 原始来源响应或其保存记录；
- 外部调用尝试；
- 待处理工作；
- 摘要生成记录；
- 错误、重试时间和完成结果。

### 5.2 逻辑数据表

具体 SQL 可在实现计划中确定，但数据关系固定如下。

| 数据 | 作用 | 关键约束 |
|------|------|----------|
| `literature_topics` | 用户的关注方向 | 归属单一用户；删除或暂停不删除公共论文 |
| `literature_papers` | 公共论文身份 | 按来源与规范化论文 ID 全局唯一 |
| `literature_paper_versions` | 论文版本与版本内容 | 按论文和来源版本唯一；保存内容指纹 |
| `literature_topic_matches` | 论文匹配到哪些关注方向 | 按关注方向和论文唯一；保存匹配原因 |
| `literature_user_paper_states` | 已读、收藏、忽略等用户状态 | 按用户和论文唯一，不依赖关注方向 |
| `literature_checks` | 每日或手动检查的总体记录 | 固定检查范围和稳定去重键 |
| `literature_check_pages` | 每个分页或分片的完成情况 | 可从中断页继续；只有全部完成才算完整覆盖 |
| `literature_source_snapshots` | 原始来源响应及校验信息 | 响应成功后优先保存；允许设置保留期限 |
| `literature_api_attempts` | 每次外部调用的开始、结果与未知状态 | 失败、超时和可能已发送必须可区分 |
| `literature_summaries` | 某论文版本的摘要 | 按论文版本、摘要版本和语言唯一 |
| `literature_research_task_links` | 论文与研究任务关系 | 按用户、论文和任务唯一 |
| `literature_work_items` | 待处理、处理中和待重试工作 | 不绑定具体队列框架；可在重启后恢复 |

### 5.3 论文身份与版本

论文身份和版本必须分离。

规范化论文身份至少包含：

- 来源，例如 `arxiv`；
- 不含版本后缀的来源 ID；
- 可选 DOI。

论文版本至少包含：

- 来源版本，例如 `v1`、`v2`；
- 来源的 `updated` 时间；
- 标题和 abstract；
- 作者与分类；
- 内容指纹；
- 首次获取时间。

不能通过简单地在字符串中遇到字母 `v` 就截断版本；旧式 arXiv 分类 ID 也必须正确规范化。

### 5.4 用户论文状态

已读、收藏和忽略属于“用户与论文”，不属于“关注方向与论文”。这样一篇论文匹配多个方向时仍然只有一个用户状态。

状态包括：

- `is_read`；
- `is_saved`；
- `is_ignored`；
- `first_seen_at`；
- `last_seen_at`；
- `latest_seen_version_id`。

“新到”和“有新版本”可以根据用户已见版本与论文当前版本计算，不需要为每个关注方向复制状态。

### 5.5 摘要身份

摘要必须绑定论文版本，而不是只绑定论文 ID。

摘要唯一条件至少包含：

```text
paper_version_id
+ content_hash
+ summary_recipe_version
+ model
+ language
```

摘要状态包括：

- 等待生成；
- 正在生成；
- 已完成；
- 生成失败；
- 已过期。

论文出现新版本且内容发生变化时，旧摘要仍可保留用于历史查看，但当前版本显示为“需要更新摘要”。

### 5.6 原始响应保留

外部响应成功返回后，应优先保存原始内容或可验证的完整快照，再执行解析和本地匹配。

这样可以保证：

- 解析失败时不重新请求 arXiv；
- 程序升级后可以重新解析历史响应；
- 能够判断外部调用到底获得了什么；
- 可以核对取得数量与最终保存数量。

原始响应允许采用有限保留期限，例如 30 到 90 天；论文、版本、检查结果和外部调用记录长期保留。

## 6. 每日检查与本地缓存

### 6.1 固定检查范围

每次检查在开始前确定固定范围：

```text
window_start = 本次检查开始边界
window_end   = 本次检查固定结束边界
```

检查执行时间再长，也不能在结束时把“当前时间”直接作为下一次开始边界。只有固定范围全部完成后，才能推进覆盖位置。

建议每日检查保留 48 到 72 小时重叠，通过论文 ID 和版本在本地去重，以处理：

- 服务停机；
- arXiv 延迟收录；
- 论文发布新版本；
- 上次检查部分完成；
- 时区和计划时间波动。

### 6.2 查询覆盖缓存

系统必须区分：

- 论文缓存：本地已经有一些论文；
- 查询覆盖缓存：某个来源、条件和固定时间范围已经完整检查。

只有查询覆盖记录为完整时，才能安全跳过外部调用。

覆盖记录至少包含：

- 来源；
- 规范化条件；
- 固定开始和结束时间；
- 排序方式；
- 分页或分片信息；
- 查询构造版本；
- 是否完整；
- 完成时间。

空结果也应保存为有效覆盖结果，防止用户反复触发相同的无结果查询。

### 6.3 统一收集与本地匹配

目标架构不再为每个关注方向独立调用 arXiv。

系统优先：

1. 汇总活跃关注方向涉及的分类和时间范围；
2. 将可以复用的来源检查合并；
3. 按分类、时间范围和必要分页收集论文；
4. 在本地文献库中使用标题、abstract、分类、包含词和排除词匹配关注方向。

首轮实现允许先合并“完全相同的查询”，但长期目标是按来源和分类统一收集，再在本地匹配。用户和 API 契约不依赖内部采用哪一个阶段。

### 6.4 分页完整性

分页不能只以“已经拿到前 50 或 100 篇”为完成标准。

如果系统声称某个时间范围已经完整检查，就必须：

- 继续分页到时间边界；或
- 将范围拆成更小分片并逐片完成；或
- 明确标记结果仅为 Top N，而不是完整覆盖。

分页中断后，从最后一个已保存分页继续。已经保存的分页不重新请求。

### 6.5 外部调用额度与失败

系统把文献源检索视为更稀缺的上游能力，把 LLM 视为可以延后和降级的下游能力。

每次外部调用记录：

- 准备调用；
- 已开始；
- 已收到响应；
- 响应已持久保存；
- 成功、失败或结果未知；
- HTTP 状态和可选来源请求 ID；
- 下一次允许重试时间。

超时不能简单视为“没有调用”，因为请求可能已经到达来源。此时记录为“结果未知”，进入冷却和人工可见的诊断状态。

失败处理原则：

- 明确的限流响应：遵守来源给出的等待时间；
- 暂时网络或服务错误：有限次数重试并逐步延长等待；
- 查询参数错误或永久拒绝：停止自动重试；
- 解析失败：使用本地原始响应重新解析；
- 预算不足：等待下一预算周期，不进入快速失败循环。

## 7. 本地匹配

关注方向匹配只使用已经进入本地文献库的数据。

匹配依据包括：

- 分类范围；
- 标题；
- abstract；
- 包含关键词；
- 排除关键词；
- 后续可扩展的本地相关度规则。

每条匹配关系保存可解释原因，例如：

```text
匹配关注方向：Agent Memory
原因：标题包含 “agent memory”；abstract 包含 “long-term memory”
```

同一论文匹配多个关注方向时，文献列表聚合显示全部方向。删除关注方向只删除对应匹配关系，不删除公共论文或用户已收藏状态。

本地预览和每日匹配使用同一套规则，避免创建时预览与实际收件箱结果不一致。

## 8. 统一 API 设计

### 8.1 入口与命名

前端的唯一正式入口为：

```text
/api/literature
```

新接口统一使用：

- `topic_id`：关注方向；
- `paper_id`：规范化论文身份；
- `version_id`：论文版本；
- `check_id`：每日或手动检查；
- `summary_id`：摘要结果或请求；
- `task_id`：研究任务。

旧 `/api/literature/subscriptions` 接口在迁移期保留，但不再作为前端新功能的正式契约。

### 8.2 通用列表格式

所有分页列表统一返回：

```json
{
  "items": [],
  "total": 0,
  "next_cursor": null
}
```

列表使用稳定游标，避免在新论文不断写入时出现 offset 重复或漏项。

### 8.3 页面总览

```text
GET /api/literature/overview
```

返回：

- 上次成功检查；
- 下次计划检查；
- 当前检查状态；
- 今日新增、未读、收藏和新版本数量；
- 是否有部分完成或需要注意的问题；
- 最近一次检查的简要结果。

该接口服务于页面顶部状态区，避免前端拼接多个零散接口推断系统状态。

### 8.4 关注方向

```text
GET    /api/literature/topics
POST   /api/literature/topics
GET    /api/literature/topics/{topic_id}
PATCH  /api/literature/topics/{topic_id}
DELETE /api/literature/topics/{topic_id}
POST   /api/literature/topics/preview
```

要求：

- 所有操作只作用于当前用户；
- `preview` 不直接请求外部来源；
- 创建和更新返回关注方向以及本地覆盖说明；
- 删除默认只删除关注方向和匹配关系，不删除论文、阅读状态或任务关联；
- 暂停关注方向不会清除历史结果。

### 8.5 文献列表

```text
GET /api/literature/papers
```

主要查询条件：

```text
view=today|unread|saved|updated|all
topic_id=...
category=...
summary_status=...
has_research_task=true|false
cursor=...
limit=...
```

返回的是当前用户文献库中的论文，不暴露其他用户的关注方向或阅读状态。

每个列表项包括：

- 论文基础信息和当前版本；
- 当前用户的阅读状态；
- 匹配关注方向和原因；
- 摘要状态；
- 新到或新版本提示；
- 研究任务关联摘要。

### 8.6 文献详情与版本

```text
GET /api/literature/papers/{paper_id}
GET /api/literature/papers/{paper_id}/versions
```

访问前必须验证论文属于当前用户的文献库，或已被当前用户收藏、关联任务。不能只因为其他用户拥有该论文关联就允许操作。

### 8.7 阅读状态

```text
PATCH /api/literature/papers/{paper_id}/state
```

统一支持：

- 已读或未读；
- 收藏或取消收藏；
- 忽略或取消忽略。

状态写入当前用户与论文的唯一记录，不再依赖 `subscription_id`。这样可以从根本上避免同一论文多个关注方向之间的状态冲突和跨用户更新风险。

### 8.8 摘要

```text
GET  /api/literature/papers/{paper_id}/summary
POST /api/literature/papers/{paper_id}/summary
```

`GET` 返回当前版本的摘要状态和已有结果。

`POST` 表示请求生成当前版本摘要：

- 已有有效结果时直接返回；
- 已有相同工作时返回同一个摘要请求；
- 旧版本摘要存在时返回“当前版本需要更新”；
- 创建新工作后立即返回状态，不等待 LLM 完成。

重复请求不得重复消耗 LLM 调用。

### 8.9 研究任务

```text
POST /api/literature/papers/{paper_id}/research-task
GET  /api/literature/papers/{paper_id}/research-tasks
```

创建研究任务和保存论文关联对用户表现为一个操作。请求重复提交时，应复用已有的创建结果或返回明确的正在处理状态。

### 8.10 检查最新文献

```text
POST /api/literature/checks
GET  /api/literature/checks/current
GET  /api/literature/checks/{check_id}
GET  /api/literature/checks
```

`POST /checks` 支持可选关注方向范围，但不提供普通用户绕过缓存、覆盖证明或外部限流的强制开关。

重复调用行为：

- 相同检查正在进行：返回现有 `check_id`；
- 最近覆盖完整且仍有效：返回缓存结果；
- 仅部分覆盖：安排缺失部分；
- 来源处于等待期：返回当前状态和预计重试时间。

前端不再每秒固定轮询 60 次。初始版本可使用逐步放慢的状态刷新；后续可以增加服务端事件通知，但持久化检查记录始终是权威状态。

## 9. 统一状态语言

前端显示、API 返回和持久化记录使用同一组含义。

### 9.1 检查状态

| API 状态 | 用户显示 | 含义 |
|----------|----------|------|
| `planned` | 等待检查 | 已确定范围，尚未开始 |
| `checking` | 正在检查 | 正在取得或处理文献源结果 |
| `partial` | 部分完成 | 已有结果可用，但仍有缺失范围 |
| `completed` | 已完成 | 固定范围已经完整覆盖 |
| `retrying` | 稍后重试 | 暂时失败且已安排后续尝试 |
| `failed` | 无法完成 | 需要调整条件或人工处理 |

### 9.2 摘要状态

| API 状态 | 用户显示 |
|----------|----------|
| `not_requested` | 尚未生成摘要 |
| `queued` | 等待生成 |
| `generating` | 正在生成 |
| `completed` | 摘要已完成 |
| `failed` | 生成失败，可重试 |
| `stale` | 论文已更新，需要更新摘要 |

### 9.3 关注方向状态

| API 状态 | 用户显示 |
|----------|----------|
| `pending_first_check` | 等待首次检查 |
| `active` | 正常关注 |
| `paused` | 已暂停 |
| `attention_needed` | 需要调整条件 |

## 10. 模块职责

具体文件拆分由实现计划确定，但业务职责固定为以下模块。

### 10.1 关注方向

负责创建、编辑、暂停、删除和本地预览，不直接调用文献源。

### 10.2 每日检查

负责确定需要检查的范围、合并重复需求、跟踪分页和判断覆盖是否完整。

### 10.3 文献源访问

只负责按已经确定的请求取得来源响应，并记录调用结果。它不生成摘要、不修改用户阅读状态。

### 10.4 本地文献库

负责论文身份、版本、原始来源快照和公共元数据的持久保存。

### 10.5 本地匹配

负责将论文匹配到关注方向，并保存匹配原因。预览和每日匹配共用同一规则。

### 10.6 用户收件箱

负责新到、未读、收藏、忽略和有新版本等用户视图，不负责外部抓取。

### 10.7 阅读辅助

负责摘要、翻译和后续阅读能力。所有结果按论文版本保存，失败可以单独恢复。

### 10.8 研究任务转化

负责创建任务并保存论文关联，保证用户不会遇到只完成一半的状态。

### 10.9 统一 API

负责身份校验、用户权限、输入输出和对上述模块的组合，不保存仅存在于 API 进程内的长期任务状态。

## 11. 一致性与恢复原则

### 11.1 分阶段提交

文献处理拆成可独立完成的阶段：

```text
来源响应已保存
  → 论文版本已保存
  → 本地匹配已完成
  → 用户收件箱已更新
  → 摘要已生成
  → 研究任务已关联
```

每个阶段成功后立即保存。后续阶段失败不会撤销前面的有效结果。

### 11.2 上游结果不可被下游失败浪费

- LLM 失败不重新请求 arXiv；
- 本地匹配失败不重新请求 arXiv；
- 页面状态更新失败不重新请求 arXiv；
- 研究任务创建失败不重新生成摘要；
- 解析失败优先使用已保存原始响应重新处理。

### 11.3 重复请求复用已有工作

以下行为必须可重复调用且不制造重复工作：

- 相同时间范围的文献检查；
- 同一论文版本的摘要请求；
- 同一论文创建研究任务的重复提交；
- 同一关注方向与论文的匹配；
- 同一用户与论文的状态更新。

### 11.4 运行进程不是权威状态

进程内锁、缓存和正在运行集合只用于加速。重启后，系统必须能从数据库判断：

- 哪些检查未完成；
- 哪些分页需要继续；
- 哪些外部调用结果未知；
- 哪些摘要等待重试；
- 哪些研究任务关联需要补完。

## 12. 权限与多用户边界

- 公共论文和公共摘要可以复用，但用户关注方向、阅读状态、匹配关系和任务关联严格隔离。
- 更新阅读状态必须同时验证当前用户和论文访问关系，不能根据论文被任意用户拥有就允许修改。
- 关注方向 ID 不能用于修改其他用户的匹配关系。
- 用户删除关注方向不会改变其他用户的数据。
- 后台统一收集可以共享来源调用，但结果分发必须按用户重新建立清晰关系。
- API 返回论文时必须给出当前用户的状态，不返回不明确的 `subscription_id`。

## 13. 参考项目的采用与取舍

参考 [colorfulandcjy0806/Arxiv-tracker](https://github.com/colorfulandcjy0806/Arxiv-tracker) 的每日检索设计，采用以下思路：

- 固定每日运行时间；
- 防止同一每日任务并发执行；
- 使用新鲜度窗口和重叠范围；
- 在摘要前过滤已见论文；
- 通过分页找到真正的新论文；
- 成功输出后再更新用户可见的投递状态。

不直接采用：

- 将检索、摘要、网页和邮件串成一次单体运行；
- 使用 `seen.json` 作为长期业务状态；
- 新运行取消正在进行的有效运行；
- 默认抓取论文 HTML 或 PDF；
- 把 Git commit 当作业务状态持久化；
- 在没有统一预算和结果检查时进行高次数整体重试。

OpenScience 需要进一步区分“已取得”“已匹配”“已进入收件箱”“已摘要”和“已转任务”。

## 14. 迁移策略

### 14.1 API 迁移

1. 新增 `/api/literature/topics`、`papers`、`checks` 和按需摘要接口。
2. 前端切换到新接口和新状态语言。
3. 旧 `/subscriptions`、`fetch-status`、`read` 和 `convert` 接口保留短期兼容。
4. 兼容接口内部调用新服务，不再维护第二套业务逻辑。
5. 完成前端与外部调用方迁移后，再删除旧接口。

### 14.2 数据迁移

- 旧 `literature_subscriptions` 迁移为关注方向，保留原 ID 作为兼容映射。
- 旧论文主表继续作为公共论文基础，补充规范化来源身份与版本表。
- 当前订阅—论文关联迁移为关注方向匹配关系。
- 同一用户、同一论文在多个旧订阅下的阅读状态合并为用户论文状态：任一关联已读则视为已读。
- 已转换任务的 `task_id` 迁移为研究任务关联，不因关注方向删除而丢失。
- 已有摘要迁移到摘要表；无法确定对应内容版本的摘要标记为 legacy 或需要更新。
- 旧 `last_fetched_at` 只作为迁移参考，不能直接声明某个时间范围已经完整覆盖。

### 14.3 UI 迁移

1. 先提供新总览和统一论文列表，保留旧关注方向管理入口。
2. 再将关注方向管理移到独立视图，并移除常驻左侧订阅栏。
3. 将论文卡片状态改为用户级状态和多关注方向标签。
4. 将摘要改为按需状态。
5. 最后增加更新记录和部分完成体验。

## 15. 可观测性与用户反馈

后台指标不仅统计请求成功率，还要统计是否产生有效结果。

关键指标：

- 外部来源请求数；
- 每个有效新论文需要的来源请求数；
- 查询覆盖缓存命中率；
- 被合并的重复检查数量；
- 分页恢复数量；
- 来源响应成功但未持久化的数量；
- 结果未知的外部调用数量；
- 新论文进入收件箱的延迟；
- 摘要请求、缓存命中、失败和过期数量；
- 未完成工作最久等待时间。

用户提示必须优先说明影响，而不是暴露内部异常：

```text
今天的检查已完成，新增 12 篇论文。
```

```text
今天的检查已获得部分结果，其余内容将在稍后继续；当前结果可以正常阅读。
```

```text
arXiv 暂时不可用，页面仍显示上次成功结果，系统会稍后重试。
```

## 16. 测试与验收

### 16.1 数据与权限

- 同一论文匹配多个关注方向时只保存一份公共论文。
- 两个用户对同一论文的已读、收藏和忽略状态互不影响。
- 删除关注方向不会删除论文、用户收藏或研究任务关联。
- 新论文版本可以与旧版本并存，当前版本正确更新。
- 摘要绑定正确版本，新版本出现后旧摘要显示为过期。

### 16.2 检查与恢复

- 两次相同手动检查返回同一个检查，不重复调用来源。
- 来源响应保存后进程退出，重启后从本地响应继续解析。
- 分页中断后只请求缺失分页。
- LLM 失败后不会重新调用 arXiv。
- 本地匹配失败后不会重新调用 arXiv。
- 固定检查范围完整完成后才推进覆盖位置。
- 空结果被保存为有效覆盖，不会在短期内反复请求。

### 16.3 API

- `/overview` 能表达完成、检查中、部分完成和失败状态。
- 关注方向预览只使用本地文献库。
- 论文列表分页稳定并返回总数。
- 论文详情包含版本、关注方向、用户状态和摘要状态。
- 重复摘要请求复用同一个结果或处理中记录。
- 创建研究任务和论文关联对用户表现为一个完整动作。
- 所有接口验证当前用户权限。

### 16.4 UI

- `/literature` 默认打开今日更新。
- 没有新论文、正在检查和检查失败具有不同空状态。
- 一篇论文显示多个关注方向标签而不重复出现。
- 摘要未生成时论文仍可正常阅读和操作。
- 页面只有一个明确的“检查最新文献”动作。
- 关注方向管理不再长期占据文献列表主界面。

### 16.5 最终验收标准

1. 所有文献业务状态都有持久记录，API 进程重启不会丢失。
2. 同一来源检查、同一论文版本摘要和同一研究任务创建具备重复保护。
3. arXiv 成功结果在任何 LLM 调用之前持久保存。
4. 每日收件箱可以在没有摘要的情况下及时显示论文。
5. 用户能够理解上次检查、当前检查和下次检查状态。
6. 同一论文跨关注方向聚合显示，用户状态不再依赖关注方向 ID。
7. 新旧 API 和数据迁移有明确过渡期，不要求一次性替换全部实现。

## 17. 后续设计决策

完成本文后，下一阶段需要单独讨论和决定：

1. 后台工作首先采用数据库持久队列，还是直接增加专用消息服务；
2. 文献源统一收集从“相同查询合并”还是“按分类每日收集”起步；
3. 本地关键词匹配的首版规则与后续全文检索方式；
4. 原始来源响应的存储形式和保留期限；
5. 自动摘要是否只支持 Top N，以及 Top N 的相关度判断方式；
6. 每日检查的默认时间、重叠范围和失败补偿上限；
7. staging 和 production 后台工作进程的资源与额度边界。

上述选型不得改变本文确定的用户体验、持久化数据、统一 API 和分阶段恢复原则。
