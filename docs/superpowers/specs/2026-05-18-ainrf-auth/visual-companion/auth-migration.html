多用户鉴权 — Migration & Testing

Section 4/5 · 从单 API Key 切换到 JWT + 初始 Admin 创建 + 测试策略

1. 初始 Admin 创建路径

首次启动时：
# 如果 users 表为空，创建默认 admin
def _ensure_admin_user():
    if count_users() == 0:
        password = secrets.token_hex(12)  # 随机 24-char 密码
        create_user(username="admin",
                    display_name="Administrator",
                    password=password,
                    role="admin",
                    status="active")
        logger.info(f"Created admin user. Password: {password}")
        # 密码打印到终端 stdout
也支持 ainrf admin create <username> --role admin CLI 命令。`ainrf onboard` 在首次部署时引导创建首个 admin。

2. 迁移路径（单用户 → 多用户）

现有用户（旧 API Key 部署）：
  1. 升级 ainrf 包 → `ainrf serve` 自动创建 auth.sqlite3 + 首次 admin
  2. 旧 `X-API-Key` middleware → 移除。JWT middleware 替代
  3. 所有现有 task/project/workspace/session 数据 → owner_user_id = NULL
  4. NULL owner 的资源 → 仅 Admin 可见（需要 Admin 手动重新分配，或保留 "Legacy" 视图）
⚠ Breaking change: 旧 API Key 不再工作。升级后必须创建用户并通过 JWT 认证。CI/CD 或自动化脚本需要用 ainrf login 获取 token。

3. 测试策略

Backend (pytest)

  • User 注册/登录/refresh/logout
  • JWT 签发与验证（过期/无效/篡改）
  • 密码 bcrypt hash 正确性
  • Admin 审批激活/禁用用户
  • require_admin 403 检查
  • check_project_access（owner/collaborator/none）
  • check_environment_access
  • 资源 List 按权限过滤
  • 非 owner 无法修改/删除他人资源
  • Admin 可查看/修改所有资源
  • NULL owner 资源仅 Admin 可见
  • Collaborator member 可写/viewer 不可写

Frontend (Vitest)

  • LoginPage 渲染 + 表单提交
  • RegisterPage 渲染 + pending 提示
  • AuthContext: 登录态持久化、logout 清空
  • 未登录重定向到 /login
  • Pending 用户看到等待审批页面
  • Admin Settings → Users tab 渲染
  • Admin 无法操作的按钮隐藏
  • Project Collaborators UI

4. Implementation Order

Step 1 auth.sqlite3 + User 表 + AuthService（register/login/refresh/me）
Step 2 JWT middleware 替换 API Key middleware + app.py 集成
Step 3 资源表 ALTER TABLE 加 owner_user_id + project_collaborators + environment_access 表
Step 4 permissions.py 权限检查函数 + 各 route handler 接入
Step 5 Admin API + Admin Settings 前端（Users tab, env 权限组）
Step 6 前端 LoginPage + AuthContext + Layout 保护 + CLi login
Step 7 Collaborators UI + 集成测试 + 端到端验证

5. 这是一个大功能

建议拆分为 2-3 个子 project，分别 brainstorm / spec / plan / implement

比如：Phase A（User 系统 + JWT + Login）→ Phase B（资源隔离 + 权限）→ Phase C（Admin UI + Collaborators）

单个 monolith spec 太大，容易遗漏细节。你同意这个拆分？
