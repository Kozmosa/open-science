AINRF 多用户鉴权 — 整体架构方案

基于已确认的需求：JWT + 用户名密码 + Owner/Collaborator/Admin + 注册审批

需求总结

维度 选择
部署规模小团队共享（2-10 人，同一台机器）
认证用户名 + 密码 → JWT（废弃 API Key）
权限模型Owner-based + Project Collaborators (member/viewer) + Admin(PI)
EnvironmentsAdmin 管理 + 权限组 + 资源配额
Workspaces用户隔离（默认）+ Project 内共享（collaborator 可访问）
用户创建开放注册 + Admin 审批启用
认证方式统一 JWT（WebUI + CLI 通用）

三种实现路径

A. 渐进式：最小迁移

保留现有 X-API-Key middleware，新增 JWT middleware 作为并行路径。User 表 + login endpoint + JWT 签发。资源表逐步加 owner_user_id 列。向前兼容：现有部署不受影响。

Pros

  • 零停机迁移，现有 API key 继续工作
  • 可以逐步切换资源表，每步独立上线
  • 风险最低，每步可回滚

Cons

  • 两套鉴权并存期长，代码有分支
  • JWT + API Key 双路径增加 middleware 复杂度
  • owner_user_id 逐步加，过渡期数据可能不一致

B. 硬切换：一次性替换 （推荐）

一次性替换 X-API-Key middleware → JWT middleware。User 表 + 所有资源表加 owner_user_id。Admin 初始账号通过 CLI 创建。一次性 build，一次性 migration，一次性切。

Pros

  • 架构干净，无双路径遗留
  • 所有资源从第一天就有 owner，无过渡态
  • middleware 逻辑简单：只验 JWT

Cons

  • 一次性改动大（~15 files），测试工作量大
  • 现有部署升级需要重新配置用户
  • 如果出问题回滚代价高

C. 拆分为两个 Phase

Phase 1: User 系统（注册/登录/JWT/User表）+ 替换 middleware，但不加资源隔离。Phase 2: 逐资源加 owner_user_id + 权限检查 + 前端改造。两个 phase 独立上线。

Pros

  • 每个 phase 范围可控，不互相阻塞
  • Phase 1 可以快速上线（login 页面 + JWT）
  • Phase 2 可以按资源优先级分批做

Cons

  • Phase 1 上线后到 Phase 2 之间，数据仍是全局可见
  • 两个 phase 两次大改 middleware 和前端
  • Phase 1 不提供实际的多用户隔离价值

推荐 B — 一次性替换

AINRF 当前是单用户工具，用户量少。没有遗留数据迁移的包袱。一次性做干净，比分 phase 更省总工作量。关键风险通过 充分的集成测试 覆盖。
