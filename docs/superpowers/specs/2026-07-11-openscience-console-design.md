## 1. 背景与结论

OpenScience 已经具备一个可用的前端基础：共享设计系统、严格 TypeScript、按需加载、统一 query key，以及 Tasks、Literature、Resources 等核心页面。现有问题不在于缺少新的视觉概念，而在于不同页面和早期组件对同一概念给出了不同答案：留白、背景、导航密度、操作层级和状态反馈尚未收敛。

本设计不把 WebUI 重做成营销网站，也不复制 Apple 的品牌元素。它采用的方向是：

> **OpenScience Console** 是一套安静、精确、连贯的研究工作台界面；它借鉴 Apple 对留白、层级、排版和动作层级的克制处理，同时保持科研工作所需的信息密度与可操作性。

`Prism Console` 可作为历史和内部设计语言代号，但不能作为用户可见产品品牌。用户界面主名称是 **OpenScience**；`osci` 是官方紧凑缩写和设计系统命名空间。

## 2. 设计目标与非目标

### 2.1 目标

1. 用户在任一页面能快速回答三个问题：这里是什么、目前状态如何、下一步最重要的动作是什么。
2. 所有页面共享纯白画布、相同的页头节奏、操作层级和状态语言，不再出现内容区与边缘留白底色割裂。
3. 让文献、任务、工作区和资源四类高频工作面拥有一致的外壳，但不强行采用同一种卡片或分栏布局。
4. 用低噪声的视觉语言表达状态：蓝色表示用户主要动作，语义色只表达工作结果或风险。
5. 保持键盘可用、窄屏可用、低性能设备可用；视觉效果不能依赖模糊、动画或外部字体下载才能成立。

### 2.2 非目标

- 不模仿 Apple 的标志、图形、文案、导航结构或营销页面布局。
- 不把所有页面改成低密度的宣传海报，也不把每条数据都变成大卡片。
- 不以重命名目录、迁移 feature 文件夹或替换技术栈作为本次设计工作的验收目标；这些只在能改善具体体验或维护成本时实施。
- 不在首轮引入 Storybook、视觉回归云服务或新的大型 UI 组件库。它们可在设计和测试边界稳定后单独评估。

## 3. 已确认的品牌与命名规则

### 3.1 产品名称

- 面向用户的完整名称：`OpenScience`。
- 官方缩写：`osci`。它用于紧凑标识、技术语境、设计系统命名空间和文档，不替代常规界面中的完整产品名称。
- `ainrf` 只保留在必要的运行时兼容路径、数据字段和迁移层；它不能出现在新的用户界面文案、样式或组件名称中。

### 3.2 设计系统命名空间

所有新建的 OpenScience 设计系统自定义 CSS 标识必须使用小写 `osci-` 前缀：

- 类名：`osci-topbar`、`osci-page-header`；
- 自定义属性：`--osci-surface-glass`、`--osci-color-primary`；
- 数据属性：`data-osci-variant`。

React 组件仍使用语义名称，如 `TopBar`、`PageHeader`、`StatusBadge`，不人为添加前缀。现有 `--prism-*`、`--apple-*` 和早期语义别名仅是迁移兼容层；新代码不得继续新增这些名称。

该规则只约束自定义属性、自定义全局类和数据属性。Tailwind 工具类、第三方组件类名和组件内部临时组合类不添加 `osci-` 前缀。本地存储使用 `openscience:*` 命名空间。

### 3.3 品牌标识

首版采用已确认的原创、矢量优先标识：**Open Orbit**。它由一个在右上方打开的蓝色环和一个独立的轨道点构成：

- 打开的圆环表示开放的研究入口，而不是封闭的平台；
- 轨道点表示可被追踪、验证和延展的研究发现；
- 圆形轮廓在小尺寸下仍可识别，适用于侧栏收起态、浏览器图标和命令面板；
- 标识默认使用 OpenScience 蓝，也必须支持单色版本；不能依赖渐变、阴影或位图纹理。

第一版矢量源文件为 [`frontend/public/openscience-mark.svg`](../../../frontend/public/openscience-mark.svg)。展开侧栏显示图形加 `OpenScience` 文字；收起时只显示图形，`osci` 仅作为无图形时的文字回退，不替代标识本身。

## 4. 视觉语言

### 4.1 画布、层级与密度

- 应用主画布、页面留白与主要内容面均为纯白。页面不能通过“外围白、内容灰”的背景差制造层级。
- 局部信息块可使用极浅中性或浅蓝色背景，但只用于摘要、状态、筛选、空状态和提示，不能扩张成整页底色。
- 内容层级优先由排版、留白、1px 极淡边界和必要时的微弱阴影表达；避免厚边框、深阴影和大面积玻璃卡片。
- 默认工作密度为“可扫读、可操作”，不是营销页的大留白。只有首次使用、关键空状态或单一重点结果允许采用更大的视觉留白。

### 4.2 色彩和操作层级

- OpenScience 蓝是唯一品牌主色。它用于主按钮、当前导航、焦点、链接和有限的选中态。
- 成功、等待/警告、失败分别使用绿色、琥珀色、红色；它们只表达语义，不能承担导航或产品装饰职能。
- 每个页头最多一个实心主按钮。次要动作使用描边按钮、文字按钮或收纳到菜单中。
- 所有状态颜色必须同时提供文字、图标或明确标签，不能仅依赖颜色区分。

### 4.3 排版

- 中文正文以系统字体栈为可靠基底；不让外部网络字体成为中文可读性的前提。
- 英文、数字和代码可使用现有显示/正文/等宽字体策略，但必须与中文字体的字重、行高和数字宽度协调。
- 页标题用于页面定位，不应模拟官网 Hero 的巨大标题。工作页标题保持紧凑；只有欢迎、空状态或单一结果页可使用更突出的标题层级。

### 4.4 动效

- 仅用动画表达层级变化和操作反馈：展开/收起、抽屉进入、选中态改变、后台状态完成。
- 不使用持续循环、纯装饰或同时在多个区域竞争注意力的动画。
- 必须尊重 `prefers-reduced-motion`；关闭或缩短动画后，信息层级和操作结果仍应完整可辨。

### 4.5 主题策略

设计系统必须明确提供浅色与深色两套官方主题，并通过相同的语义 token 契约支持更多官方配色和受控第三方主题。OpenScience 默认使用浅色主题；是否在第一批界面中开放主题选择器，由两套主题完成浏览器验收后决定。主题不得依赖组件中的散落 `dark:` 分支或自动反色，完整规范见关联的 `osci` 设计系统说明。

## 5. 全局外壳

### 5.0 桌面与窄屏结构

```text
┌──────────────────┬─────────────────────────────────────────────┐
│ OpenScience      │ TopBar: [侧栏] [命令面板]      [任务] [语言] │
│                  ├─────────────────────────────────────────────┤
│ 研究             │ PageHeader                                  │
│  今日概览        │ 标题、状态/说明、页面主操作                  │
│  Projects        ├─────────────────────────────────────────────┤
│  Tasks           │                                             │
│  Literature      │ 可滚动页面内容                              │
│  Workspaces      │                                             │
│                  │                                             │
│ 运行             │                                             │
│  ...             │                                             │
│                  │                                             │
│ Settings         │                                             │
│ [头像] 用户      │                                             │
└──────────────────┴─────────────────────────────────────────────┘
```

- 宽屏默认展开侧栏，建议宽度 240px；用户折叠后为 56px，并按用户 ID 保存偏好。
- 中等宽度默认折叠；窄屏侧栏变为临时 Sheet，不长期占用内容宽度。
- 收起侧栏后只显示图标，分组用细分隔线表示，所有入口提供 tooltip 和可访问名称。
- TopBar 固定 48px；PageHeader 和正文位于统一页面滚动区。TopBar 覆盖滚动内容时产生毛玻璃效果。
- 页面外壳不负责业务数据滚动。Tasks、Literature、Resources 等页面可以在外壳内部定义自己的列表或分栏滚动区，但必须保持 PageHeader 可定位。

### 5.1 左侧导航

OpenScience 是桌面研究工作台，保留左侧导航而不是照搬 Apple 官网的横向商品导航。导航应保持安静：

- 展开状态显示图标、名称和必要时的一行短说明；收起状态保留图标与可访问名称。
- 当前项使用浅蓝选中底与蓝色前景；普通项以中性文字呈现。
- 导航中的项目必须按用户目标而不是实现模块组织。管理员专有工具必须与普通研究工作分隔。
- 不把任务运行统计、说明文字或所有二级入口长期堆进导航；它们应进入页面状态区、二级导航或按需展开的菜单。

当前 `Projects`、`Terminal`、`Tasks`、`Workspaces`、文件浏览、`Environments`、`Resources`、`Literature`、`Sessions`、`Timeline` 和 `Settings` 平铺于导航中。这是本设计需要重新分组的主要对象。

已确认的首版分组如下；分组只使用小标签与细分隔线，不使用下拉菜单或可折叠的“文件夹式”导航：

```text
研究
  今日概览
  Projects
  Tasks
  Literature
  Workspaces

运行
  Terminal
  Environments
  Resources

管理（仅管理员）
  Sessions
  Timeline

固定在底部
  Settings
```

文件浏览不再作为一级导航；它从 Workspace 的项目、文件或最近位置进入，深链接仍须继续可用。Resources 对普通研究者保持可见；以后可按用户与环境关系收窄其可见数据，但不在本轮隐藏入口。Projects 保持一等入口，其与 Tasks、Workspaces 的职责、权限和生命周期统一遵守 Project / Task / Workspace 领域设计，视觉改造不得擅自降级、合并或创造另一套关系。

账户入口位于侧栏底部，使用头像和显示名称；点击后打开账户相关操作与退出菜单。侧栏顶部不再长期占用一行用户名和退出按钮。

### 5.2 TopBar

TopBar 是应用内容区的固定顶部栏，只负责全局低频操作；它不承载页面标题、页面主操作或账户入口。

- 桌面端高度采用 48px，窄屏可增加触控空间。
- 常态为接近白色的半透明表面；页面内容在其后滚动时采用毛玻璃效果，仍保留不支持时的实色回退。
- 已确认的参考参数为：实色回退约 `rgba(255, 255, 255, 0.92)`；支持浏览器使用约 `rgba(255, 255, 255, 0.72)`、`saturate(140%) blur(16px)`。它们作为 `--osci-*` token 暴露，实际值可经视觉验证微调。
- 模糊只用于 TopBar 和少量临时浮层，禁止扩散到侧栏、普通卡片和嵌套面板。
- TopBar 必须位于实际滚动内容之上或与其处于同一滚动语境，否则模糊会退化为没有层次感的半透明白条。
- 左侧依次放置侧栏展开/收起按钮和 Command Palette 入口；右侧放置紧凑的运行任务提示与语言切换。账户入口只位于侧栏底部。

### 5.3 PageHeader 与页面状态区

每个一级页面使用统一的页面头部结构：

```text
页面标题 + 一句简短状态/目的                         一个主操作
可选：本页筛选、视图切换、最近更新时间或工作状态
```

PageHeader 是页面标题的唯一归属。不要在 TopBar、侧栏和正文中重复同一个页面标题或主要动作。后台检查、加载、部分完成和失败等状态应靠近它影响的页面对象展示，而不是只在全局栏显示。

### 5.4 Command Palette

OpenScience 提供一个类似 macOS Spotlight、Listary 与 VS Code Command Palette 的极简命令面板，但不复制其商标、图形或完整实现。

- 快捷键：Windows/Linux 使用 `Ctrl+Shift+P`，macOS 使用 `Cmd+Shift+P`；`Esc` 关闭，方向键选择，`Enter` 执行。
- 首版只支持“前往页面”命令：今日概览、Projects、Tasks、Literature、Workspaces、Terminal、Environments、Resources、Settings；管理员额外看到 Sessions 与 Timeline。
- 输入框自动获得焦点，结果按“页面”分组，提供图标、名称和一句用途说明；不伪装成尚不存在的全文搜索。命令名称、描述和分组随当前界面语言切换，同时保留英文页面关键词匹配，方便技术用户直接输入路由或产品术语。
- 面板居中靠近视窗上方，宽度受限，使用 `--osci-surface-glass`、弱背景遮罩和明确的键盘焦点；视觉上取其安静和即时，不复制 macOS 的品牌细节。
- 后续可扩展为对象搜索和有限命令，但必须先有可解释、可授权的真实动作，不允许把任意后台操作直接暴露为命令。

首版视觉尺寸：桌面宽度约 640px，最大高度约 420px，距离视窗顶部约 16%–20%；输入区高度约 52px，结果行不低于 44px。窄屏使用左右 12px 安全边距，不强制铺满全屏。无匹配结果时显示简短说明，不显示空白列表。

## 6. 今日概览与三种页面原型

后续页面设计不逐页从零开始，而以以下三种原型派生。

### 6.0 今日概览：登录后的默认入口

`/` 在用户没有显式个人偏好时进入“今日概览”。它不是旧版技术健康页，也不是充满小组件的通用 Dashboard；它回答的是“今天我的研究工作最需要什么关注”。“今日概览”成为设置中的默认入口值，但用户仍可在个人偏好中切换到 Projects、Tasks、Terminal、Workspaces 或 Environments。

首期采用类似 Resources 的可拖拽卡片布局，而不是固定的仪表盘栅格。每个卡片只可通过可见的拖拽把手调整顺序，卡片内容内的列表、链接和按钮不触发拖拽；卡片尺寸暂不允许用户调整。顺序保存为用户本地偏好，存储键必须包含当前用户 ID，避免共享浏览器中的账户串用。拖拽排序属于可选个性化能力，首期不支持键盘调整顺序；键盘用户仍按默认或已保存顺序获得全部内容和操作。默认卡片顺序仍以行动优先级排列：

1. **需要注意**：仅在存在任务失败、等待用户处理、文献检查部分完成或环境不可用时显示；正常时完全不占用首屏。
2. **正在推进**：最多显示近期运行、等待或需要恢复的任务，并直接进入任务详情。
3. **今日文献**：显示少量未读或有新版本的高相关论文，进入完整收件箱处理。
4. **继续研究**：显示最近活动的 Projects 与 Workspaces，帮助用户回到上下文，而不是把项目统计做成装饰数字。
5. **运行状况**：只显示可行动的环境可用性或风险；完整指标留在 Resources。

页头默认不设置一个与各工作区竞争的主按钮。创建任务、检查最新文献等动作保留在其所属模块的状态区；概览只提供快速跳转和恢复工作。若“需要注意”卡片存在，它默认位于第一个位置；用户可以重排其他卡片，但该卡片不能被隐藏。

新用户或尚无研究数据时，概览只显示一个清晰的起步卡：建议先创建 Project 或 Task，并说明 Literature 与 Workspace 会在相应数据产生后进入概览。它不显示没有意义的零值指标或一组空白卡片。

### 6.0.1 数据快照与更新时间

今日概览不是实时监控页面，不进行高频自动轮询。系统每天上海时间 06:00 生成一次用户级概览快照并持久保存；页面读取最近一次成功快照，每张卡片都显示其数据截止时间。

- “需要注意”聚合任务失败或等待处理、文献检查部分完成或失败、环境不可用等可行动状态。
- “正在推进”优先显示运行中、排队、暂停或最近更新的任务，最多五项。
- “今日文献”优先显示未读和有新版本的高相关论文，最多五项。
- “继续研究”按最近活动时间显示 Projects 与 Workspaces，最多四项。
- “运行状况”只显示环境可用性和需要介入的问题，不复制 Resources 的完整指标。

06:00 更新失败时继续展示最近一次成功快照，并明确标记“数据可能已过期”、上次成功时间和失败状态。系统采用有界重试，不通过前端轮询反复触发更新。页头提供低强调度的“立即刷新”动作，允许用户显式生成新快照；重复请求必须复用同一个进行中工作。

### 6.0.2 页面布局

宽屏使用 3 列栅格，卡片携带固定跨列规则；用户只能调整顺序，不能改变跨列或高度策略：

```text
┌──────────────────────────────────────────────────────────────┐
│ 今日概览                         更新于 06:00   [立即刷新]   │
├──────────────────────────────────────────────────────────────┤
│ 需要注意（存在时全宽；不存在时不占空间）                    │
├────────────────────────────────────────┬─────────────────────┤
│ 正在推进（跨 2 列）                    │ 今日文献（1 列）     │
├────────────────────────────────────────┼─────────────────────┤
│ 继续研究（跨 2 列）                    │ 运行状况（1 列）     │
└────────────────────────────────────────┴─────────────────────┘
```

- 中等宽度变为 2 列，窄屏变为 1 列；卡片内容不横向滚动。
- 每张卡片头部包含标题、必要的数量/状态、数据截止时间和拖拽把手。
- 卡片主体最多展示少量可行动条目；卡片底部提供“查看全部”进入所属模块。
- 卡片自身加载或生成失败时保留上一次成功内容，并在卡片头部显示过期或失败状态；一个卡片失败不能阻塞整页。
- “立即刷新”是次要按钮。刷新进行中时显示统一进度并禁止重复提交，不把页面变成持续轮询状态。

### 6.0.3 概览接口契约

为了避免浏览器进入概览时并行拉取多个业务模块，后端提供用户级概览快照接口组：

- 查询当前快照：返回快照生成时间、下一次计划时间、整体状态和各卡片数据；
- 请求立即刷新：创建或复用一个刷新工作，返回稳定工作 ID；
- 查询刷新状态：只在用户显式刷新期间使用递减频率的有界轮询，完成或失败后停止。

快照是概览的展示缓存，不替代 Tasks、Literature、Projects、Workspaces 或 Resources 的权威数据。用户进入所属模块后仍读取该模块自身接口。每天 06:00 的生成工作按 `Asia/Shanghai` 解释时区，重启或失败后可以从持久状态恢复。概览生成和“立即刷新”都只能读取各领域已经持久化的数据或已有本地缓存，不得触发 arXiv、其他文献源、LLM 等昂贵外部调用，也不得绕过 Literature planner 的预算和限流。

### 6.1 收件箱型：Literature

目的：帮助用户判断“今天有哪些值得处理的信息”。

- 顶部呈现上次检查、下一次检查、当前检查状态和唯一动作“检查最新文献”。
- 主体是可筛选、可扫读的论文列表；同一论文的多个方向命中聚合显示。
- 详情、摘要和转研究任务是按需展开的上下文操作，不挤占列表首屏。

#### 6.1.1 页面结构

```text
PageHeader
  文献追踪 + 检查状态说明                   [检查最新文献]
  [管理关注方向]

UpdateStrip
  上次检查 · 下次检查 · 未读 · 新版本 · 当前状态

ViewToolbar
  今日更新 | 未读 | 收藏 | 新版本 | 全部    [方向] [分类] [更多筛选]

PaperInbox
  紧凑论文条目
  紧凑论文条目
  ...

DetailDrawer（按需）
```

- PageHeader 的唯一主操作是“检查最新文献”；“管理关注方向”使用次要按钮或文字入口。
- UpdateStrip 是单行或可换行的轻量状态区，不使用占据大量首屏的统计卡。`partial`、`retrying`、`failed` 在该区域就地解释。
- ViewToolbar 在列表滚动时可以保持可见；筛选条件写入 URL，刷新和分享链接后保持一致。
- 关注方向管理是 Literature 内的独立视图。进入该视图后，PageHeader 主操作切换为“新增关注方向”。

#### 6.1.2 论文条目与详情

论文列表采用列表与卡片之间的紧凑表面：

- 标题最多两行，作者和发布日期一行，摘要预览最多两行；
- 展示主要分类、未读/新版本状态和最多两个匹配方向，多余方向显示 `+N`；
- 收藏、已读和更多操作放在条目右侧或溢出菜单，不能依赖 hover 才可发现；
- 点击条目打开 DetailDrawer，不自动改变已读状态；已读由用户显式操作；
- 已忽略论文默认从普通视图移除，但可以通过筛选恢复查看。

DetailDrawer 展示完整摘要、全部匹配方向与原因、版本历史、来源链接、按需摘要及研究任务入口。摘要生成失败只影响摘要区域，不阻止用户阅读论文信息或打开来源。

#### 6.1.3 状态设计

- 无关注方向：显示创建第一个方向的起步状态，不发起无分类的外部检索。
- 今日无新增：明确说明检查已完成但没有新增，并提供查看未读或全部文献的入口。
- 筛选无结果：保留筛选工具，提供清除筛选，不误导为系统没有论文。
- 检查进行中：主按钮进入 loading；只在进行中使用递减轮询，完成后停止。
- 部分完成/重试：保留已获得论文，显示覆盖不完整原因和后续安排。
- 列表请求失败：保持已有列表，顶部显示非阻塞错误；不清空成功缓存。

### 6.2 工作台型：Tasks 与 Workspaces

目的：帮助用户推进“正在进行的研究工作”。

- 默认采用紧凑列表与上下文详情并置的工作面。
- 元数据、历史和低频配置从主流程中移至右侧抽屉、二级标签或按需面板。
- 任务创建是清晰可见的主操作；运行中状态可见，但不能让持续轮询和色彩淹没内容。

#### 6.2.1 Tasks 页面结构

本轮只重构信息层级和交互表面；所有创建、移动、归档、Retry、Context 更新和 Workspace 选择都必须遵守已接受的 Task、Project、Workspace 领域关系。

```text
PageHeader
  Tasks + 运行/等待摘要                         [新建任务]

TaskWorkspace
  ┌───────────────┬──────────────────────────────────────────┐
  │ 搜索、排序     │ TaskHeader：标题、状态、运行控制         │
  │ 任务列表       ├──────────────────────────────────────────┤
  │               │ 消息与执行流                             │
  │               ├──────────────────────────────────────────┤
  │               │ 输入区                                   │
  └───────────────┴──────────────────────────────────────────┘
                                      DetailDrawer：元数据/历史
```

- PageHeader 的主操作固定为“新建任务”；列表栏不再重复放置大型新建按钮。
- 左侧任务列表建议宽度 300–360px，可折叠和调整宽度；折叠偏好按用户保存。
- 中央区域是唯一主要工作面。TaskHeader 显示所选任务标题、状态、暂停/恢复等直接运行控制，不重复页面标题 `Tasks`。
- 元数据默认关闭，以右侧 DetailDrawer 打开；宽屏允许临时停靠，但不能成为永久挤压消息区的第三主栏。
- 任务 URL 保留所选任务 ID和抽屉状态，刷新后回到同一上下文。

#### 6.2.2 任务列表与操作

- 每个任务条目显示标题、状态、研究员/执行配置和相对更新时间；辅助信息不超过两行。
- 状态徽标始终可见。取消、归档、重试、删除等操作进入 shadcn Dropdown Menu，不通过 hover 才出现，也不在可点击任务按钮内嵌另一个伪按钮。
- 搜索、排序、归档筛选位于列表顶部的紧凑 toolbar；筛选变化不清除当前有效选择。
- 任务正在流式执行时优先使用已有任务流更新消息和状态；只在断线恢复或无流连接时采用有界刷新，避免列表和详情重复高频轮询。

#### 6.2.3 Tasks 状态设计

- 无任务：显示一个起步状态和“新建任务”，不渲染空白三栏。
- 列表加载、详情加载和消息加载相互独立；选择已有列表项时详情骨架不阻塞列表操作。
- 任务失败：在 TaskHeader 与消息尾部说明失败；Retry 始终作用于同一 Task，并根据引擎能力恢复原 Attempt 或创建新的 Attempt，不能克隆成另一个 Task。已归档 Task 或 Project 必须先显式解除归档。
- 流断开：保留已有消息，显示重连状态；不能把暂时断线直接显示为任务失败。
- 无权限/任务不存在：从当前选择中移除并返回可访问列表，避免页面永久白屏。

### 6.3 观察型：Resources 与 Environments

目的：帮助用户判断“系统是否可用、哪里需要介入”。

- 首屏先显示少量可行动的结论，而不是所有指标。
- 指标详情、历史和机器级信息在需要时展开。
- 健康、等待、风险状态须使用统一的语义颜色和词汇，避免各页面自行命名。

#### 6.3.1 Resources 页面结构

```text
PageHeader
  Resources + 数据状态                       [刷新]

ResourceStatus
  在线环境数 · 异常环境数 · 最新数据时间

DraggableCardGrid
  Task Usage
  Environment System
  OpenScience Processes
  ...
```

- Resources 是接近实时的观察页面，可以在页面可见时按约 5 秒刷新资源快照；页面隐藏后暂停，恢复可见时立即刷新一次。
- PageHeader 显示最近一次成功数据时间。手动刷新是次要动作，不与自动刷新并发制造重复请求。
- 卡片顺序和侧栏偏好一样按用户 ID 隔离；尺寸固定，首期不提供键盘排序或自由缩放。
- 每个环境的 System 与 Processes 卡保持可识别关联，但允许用户调整全局顺序。

#### 6.3.2 卡片层级

- Task Usage 首先给出可解释的总量和高消耗任务，不把成本、token 和时长做成无上下文的大数字。
- System 卡优先展示环境健康，再展示 GPU、CPU、内存；磁盘不进入默认首屏。
- Processes 卡默认按 CPU 或风险排序，表格列保持稳定；窄屏允许表格局部横向滚动，但卡片本身不溢出页面。
- 正常状态使用中性文字和小状态点；只有 degraded、stale、unreachable 等需要注意的状态使用语义色。

#### 6.3.3 Resources 状态设计

- 首次加载：使用卡片骨架，避免布局在数据到达后大幅跳动。
- 暂无环境：显示连接或创建环境的说明，不显示零值 CPU/GPU 卡。
- 数据过期：保留最后快照，显示 `stale` 和截止时间；不能将旧数据伪装成实时数据。
- 单环境失败：只影响对应环境卡，其他环境继续更新。
- 全局请求失败：保留旧卡片布局和最后成功数据，提供重试，不清空页面。

## 7. 组件边界

第一批需要稳定的共享组件和模式如下。它们是体验契约，不预设具体文件移动方式。

| 组件/模式 | 职责 | 不承担的职责 |
| --- | --- | --- |
| `TopBar` | 全局定位、账户与低频工具、毛玻璃表面 | 页面主操作、页面专属筛选 |
| `PageHeader` | 标题、目的/状态、一个主操作 | 复杂任务详情、长期指标面板 |
| `ViewToolbar` | 列表视图切换、搜索、筛选与排序 | 全局导航 |
| `StatusBadge` | 统一状态文本、图标、颜色与辅助说明 | 代替页面状态区 |
| `EmptyState` | 解释空状态、下一步和可恢复动作 | 承载错误堆栈 |
| `DetailDrawer` | 不离开当前上下文查看详情与低频操作 | 长期并列的第三主栏 |
| `ConfirmDialog` | 破坏性或不可逆操作确认 | 普通表单 |

## 8. 可访问性、性能与验证

- 所有导航、抽屉、菜单和主操作必须可用键盘完成；焦点可见且不会被半透明表面吞没。
- 模糊必须有实色回退。低性能设备或不支持 `backdrop-filter` 的浏览器仍应看到清晰的白色栏。
- 不在滚动列表、数据卡片或多个嵌套面板上使用 `backdrop-filter`；仅在固定的窄 TopBar 与必要浮层上使用。
- 设计验收以真实浏览器的计算样式、窄屏行为、键盘路径和页面截图为依据，而非仅凭组件代码或静态设计稿。
- 第一批视觉回归测试应覆盖 TopBar 三种状态、白色画布一致性、三个页面原型的空/加载/错误/完成状态；具体工具选型不属于本设计的首轮范围。
- 产品以桌面研究工作为第一优先。窄屏必须可完成登录、导航、查看、确认和轻量操作；复杂分栏、批量编辑和长时间任务操作允许保留为桌面优先体验，并明确提示而不是挤压成不可用界面。
- 应用浅色和深色主题都必须完成对比度、焦点、模糊回退和数据可视化验收。Terminal 可保持专用深色表面；Monaco 和普通 Markdown 代码块跟随应用主题。

## 9. 迁移原则

1. 先建立 `osci-*` token 与共享外壳，再迁移代表页面，避免全仓库批量替换样式变量。
2. 旧 `prism` token 保持兼容，直到所有使用方迁移完成；每个新组件只能消费 `osci-*` token。
3. 第一批实现面为全局外壳、今日概览、Command Palette、Literature、Tasks 和 Resources；Workspaces 紧随其后。Projects 保持一级入口，其实施必须引用已接受的 Project / Task / Workspace 领域设计，不能被本轮视觉改造替代。
4. 导航重组、默认首页和新增命令面板已完成产品决策；实施时必须保护既有深链接、权限边界和个人默认入口兼容行为。
5. 默认入口迁移必须通过版本化设置迁移函数完成：旧用户保留已有合法选择，新用户、缺失值和无效值默认使用今日概览。迁移完成后运行时不保留散落的一次性分支。

## 10. 已确认的产品决策

1. 登录默认入口新增“今日概览”，由本设计定义其信息优先级；它是个人设置中的默认值，但保留用户切换至其他现有入口的选项。
2. 侧栏按“研究 / 运行 / 管理”轻量分组，以标签与分隔线表达，不做下拉或可折叠分组。Settings 固定在底部。
3. Projects 是一级核心入口；其与 Tasks、Workspaces 的职责边界以 Project / Task / Workspace 领域设计为准。
4. 新增 Command Palette，快捷键为 `Ctrl+Shift+P` / `Cmd+Shift+P`，首期仅实现页面切换命令。
5. Resources 对普通研究者保持可见；Sessions 与 Timeline 继续受管理员权限控制。
6. 第一批实现范围为全局外壳、今日概览、Command Palette、Literature、Tasks、Resources；Workspaces 紧随其后。
7. 设计系统明确支持浅色和深色官方主题，并为更多配色和受控第三方主题预留稳定契约；默认主题为浅色。
8. 采用原创、矢量优先的 Open Orbit 标识作为 OpenScience 首版 Logo。
9. 账户入口固定在侧栏底部；新用户的今日概览使用单一起步卡，不显示空白仪表盘。
10. 今日概览每天上海时间 06:00 生成持久化快照，不高频自动轮询；每张卡片显示数据截止时间，并支持用户显式立即刷新。
11. 卡片顺序按用户 ID 隔离保存在本地，首期不支持键盘排序或尺寸调整。
12. 第一批设置界面开放浅色、深色和跟随系统主题选择，默认浅色；第三方主题首期只交付底层契约与格式，不提供导入 UI。
13. 通用组件最终统一收敛到 `osci` 封装的 shadcn 衍生实现，OpenScience 特有工作面继续自研；迁移按组件族分批完成。

## 11. 后续仍需讨论的边界

1. Projects、Tasks、Workspaces 和首期 Project Context 已完成领域设计；复杂 memory 检索与自动候选提取留待后续专项设计。
2. Command Palette 何时扩展至对象搜索、任务创建或其他命令；首期不预留不可用入口。

## 12. 验收定义

本设计完成并不意味着所有页面视觉重做完毕。它在以下条件达成后，才可进入分批实现：

- 上述已确认决策和独立领域设计被实现计划准确引用；复杂 memory 检索和自动候选提取不得被首轮实现隐式加入；
- `osci-*` token 与 TopBar/PageHeader 的视觉和交互契约被确认；
- 三个页面原型各有一份可评审的信息结构和状态设计；
- 已识别旧提案中应保留、替代或废弃的内容；
- 有明确的分批实施顺序、页面级验收场景和回归验证方式。
