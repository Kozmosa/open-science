    AINRF 领导汇报逐字稿¶

    面向对象：学院领导，计算机领域背景，但对"AI Agent"概念不熟悉。 总时长：约 6–7 分钟（每页 45–70 秒）。 风格：尊重、平实、类比清晰，避免技术黑话，关键概念用一句话解释。

    Slide 1 — 封面（约 45 秒）¶

    各位领导好。

    今天汇报的是我们团队正在推进的一个项目，代号叫 AINRF。先给大家一个总体印象：

    我们不是在做"替代研究者的 AI"，而是在做一个能减少科研重复劳动的基础设施。

    具体来说，它是一个"控制面"——就像飞机驾驶舱里的仪表板，飞行员通过它操控飞机，而不是让仪表板代替飞行员开飞机。我们的目标，是让研究者少花时间配环境、找文件入口、盯着终端屏幕，把精力集中在真正有价值的事情上：提出问题和判断结果。

    目前我们已经交付了三个层面的东西：一个命令行工具、一个后端服务、还有一个网页界面。后面我会一页一页展开。

    Slide 2 — 为什么要做（约 60 秒）¶

    先说一下，这个项目是怎么来的。

    我们在日常科研、带研究生做实验的过程中，反复遇到四类时间浪费，我相信各位老师也深有体会。

    第一类，配环境。 新服务器、新仓库、新依赖，每次都要重新踩坑。一个论文代码 clone 下来，装依赖能装一下午。

    第二类，找入口。 仓库结构复杂，README 写得不清不楚，不知道跑哪个文件、用什么命令。学生来问，老师也得花时间翻。

    第三类，复现 baseline。 论文里的代码往往跑不通，或者跑出来的结果和论文对不上，调试成本极高。

    第四类最隐性：人工值守长任务。 模型训练、网格搜索，动辄跑几个小时甚至几天，学生得一直盯着终端，生怕断了、错了、磁盘满了。

    这四件事有一个共同点：它们不直接产生科研价值，但不得不做。

    AINRF 的目标，就是把这四类"低价值操作"的时间，压到最低。

    Slide 3 — 今天已经交付了什么（约 70 秒）¶

    好，接下来汇报一下，到今天为止，我们手里有什么。

    我把它分成三层来说，从最底层到最上层。

    最底层，是一个叫 ainrf 的命令行工具。 可以安装、可以启动服务、可以做首次初始化配置。这个层面是给技术同学用的，领导们不用关心细节，知道有这么个底座就行。

    中间层，是一个后端服务。 它已经挂上了几个核心模块：环境管理、工作区管理、终端会话、任务调度、还有代码浏览器。这些模块之间是打通的，不是各自为政的。

    最上层，是一个网页界面。 我们在浏览器里做了 Tasks 任务页、Workspaces 工作区页、Settings 设置页、Terminal 终端页、还有一个 Code-Server 代码浏览器页。也就是说，研究者打开浏览器，就能完成大部分操作，不用在命令行和编辑器之间来回切换。

    这里我想强调一个原则，也是我们团队给自己定的规矩：我们只讲"已经能启动、能观察、能回看"的东西，不画大饼、不吹未来愿景。

    目前后端测试全部通过，前端测试也全部通过，代码是真实可运行的。

    Slide 4 — 远程服务 AI Agent 如何加速科研（约 75 秒）¶

    这一页是整套汇报的核心，我稍微展开说一下。

    前面提到"Agent"，可能有些领导对这个词不太熟悉。我打个比方：Agent 在这里可以理解为一个"数字科研助理"——你告诉它要做什么，它去执行，执行过程中把每一步的进展实时汇报给你。

    具体流程是这样的：

    研究者在网页上选一个工作区（比如"交通联邦预测实验"这个项目目录），再选一个执行环境（比如"实验室 3 号服务器"），然后输入任务描述，比如"帮我把这个仓库的 baseline 跑一遍，看看结果"。

    系统收到这个请求后，会做一件关键的事：把上下文分层组合起来。 一共五层：

    第一层是系统级指令，告诉 Agent 基本的执行规范； 第二层是工作区上下文，告诉 Agent 这个项目的背景； 第三层是环境配置，告诉 Agent 在哪台机器上跑、用什么工具链； 第四层是执行风格，比如"先检查再修改"还是"直接执行"； 第五层才是用户输入的具体任务描述。

    为什么要分层？ 因为同样一句"跑 baseline"，在不同的项目、不同的机器上，含义完全不同。分层组合能保证 Agent 拿到的上下文是完整的、准确的。

    然后，系统会在本地或者通过 SSH 远程连接到实验室服务器，启动一个 Claude Code 会话去执行。执行过程中，输出会实时流回浏览器，同时全部写入数据库持久化。

    这意味着什么？ 意味着即使学生的浏览器关了、电脑睡了，回来以后可以回放完整的执行过程，就像看录像一样。我们把这个叫做"可监控、可回放"——核心转变就是从"人盯终端"解放出来。

    Slide 5 — 为什么比直接手工跑 Agent 更省时间（约 60 秒）¶

    可能有的领导会问：我自己开个 Claude Code，或者让学生直接 SSH 上去跑，不也能做吗？

    差别在于四个具体的节省：

    第一，一键启动。 我们写了一个脚本 webui.sh，一键同时拉起后端服务和前端页面。不用手动配端口、配代理、配跨域，降低了很多启动摩擦。

    第二，减少重复输入。 工作区和环境都有预设配置，常用的项目、常用的服务器，第一次配好，以后直接选就行，不用每次都重新输入。

    第三，可监控、可回放。 刚才说的 SSE 实时流加上数据库持久化，长任务不用人盯着。学生可以去吃饭、去上课，回来再看结果。

    第四，远程文件浏览。 内置了一个代码浏览器，直接在网页里翻远程服务器上的目录和文件，不用 SSH 进去再开编辑器。

    这四点加起来，省的不是"跑代码"的时间——那个时间省不了——省的是准备、监控、翻找、重复配置的时间。

    Slide 6 — 正在推进的下一层能力：Skills + ARIS 基座（约 60 秒）¶

    最后汇报一下下一步的方向。

    我们正在设计一个Skills 能力层。这个概念可以这样理解：前面说的"控制面"已经能把任务发出去、把结果收回来，但任务本身怎么做，还依赖 Agent 的通用能力。Skills 层的目标，是把科研中常见的高频操作——比如项目初始化、环境自动配置、Git 分支规范、代码提交规范——做成可复用的模板。

    打个比方，就像工厂里的"标准作业程序"（SOP），不是每次重新发明流程，而是把成熟的流程固化下来，减少重复摸索。

    需要明确的是，这一层目前处于设计推进阶段，底层控制面已经具备承载条件，但上面的 Skills 模板还需要继续打磨。

    另外，我们在 test 目录已经完成了 ARIS 基座的探索实验。ARIS 是我们团队另一个研究方向的成果，覆盖从"找研究想法"到"验证实验"到"分析结果"的完整链路。后续我们会把 ARIS 接入到 AINRF 的控制面上，作为虚拟科研助理（VSA）的核心能力之一。

    简单说：ARIS 负责"聪明地做事"，AINRF 负责"稳定地管事"，两者互补，形成一个完整的科研辅助栈。

    Slide 7 — 收尾 / 争取支持（约 45 秒）¶

    最后，我们希望争取学院继续支持这个方向的 demo 和 pilot 打磨。

    下一步有三个具体落点：

    第一，把 Skills 层真正接到现有控制面上，让它跑起来，而不是停留在设计文档里。

    第二，建立可量化的度量指标。 比如环境配置平均时间、任务执行成功率、学生使用后的主观反馈。用数据说话，而不是凭感觉。

    第三，在真实科研场景上做试点验证。 我们已经在交通联邦预测这个方向上做了一些探索，接下来会把它作为第一个 pilot 场景，验证整个流程是否真的能省时间、减少重复劳动。

    最后我想再强调一次我们的定位：

    我们的目标不是"24 小时不间断的自治科研引擎"，那种东西目前不现实，我们也不想做。我们的目标，是做一个"真正有用的科研辅助工具"——简单、可靠、能落地。

    谢谢各位领导。
